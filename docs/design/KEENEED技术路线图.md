## 一、共识要点

经过三轮深入讨论，团队在以下核心问题上已达成共识：

| 序号 | 共识事项 | 决策内容 |
|:----:|----------|----------|
| 1 | **多Agent群聊核心功能** | 这是KEENEED的差异化杀器：用户进入群聊房间，5-6个预设角色的AI专家（技术、产品、设计、运营、风控、执行）围绕用户问题展开讨论、辩论、协作，用户可旁听也可插话 |
| 2 | **聊天功能 MVP** | 实现基础文字消息收发，采用轮询方案作为过渡，暂不上已读未读和离线推送；第2-4周叠加多Agent群聊能力 |
| 3 | **首页性能优化** | 采用混合方案（SSR 首屏 + CSR 懒加载），SEO 不是主要顾虑，效果不够再考虑全量 CSR |
| 4 | **稳定性加固** | 熔断降级 + 慢查询治理，单机房高可用，不做异地多活（当前阶段过度设计） |
| 5 | **技术债务清理** | 每迭代抽取 10% 资源小步还债，新增代码强制带单元测试，不发起专门的还债战役 |

## 二、分歧与决策记录

| 分歧点 | 初始立场 | 最终决策 | 决策依据 |
|--------|----------|----------|----------|
| 异地多活 | 豆朔：认为需要 | **暂不做** | 老万：当前用户量级成本收益不匹配，先做好单机房高可用即可 |

| 聊天 MVP 范围 | 待定 | **基础文字消息 + 轮询** | 聚焦核心路径快速验证 PMF |

## 五、技术债治理规范（长期有效）

> 以下规范自本路线图发布之日起生效，所有团队成员必须遵守：

1. **新增代码规范**：所有新增的业务代码必须附带单元测试，单测覆盖率纳入 Code Review 门槛
2. **迭代资源分配**：每个迭代 Sprint 中，**强制抽取 10% 开发资源**用于技术债务清理
3. **禁止继续恶化**：禁止以“赶工期”为由合并不符合规范的代码，技术债问题必须被看见和记录
4. **定期回顾**：每两周技术周会同步技术债清理进度，确保持续推进

## 八、多Agent群聊功能详细设计

> **这是KEENEED平台的核心差异化功能**
> 
> 用户不是在跟一个AI聊天，而是加入一个由5-6位AI专家组成的团队讨论会。

### 8.1 预设专家团队角色

| 角色 | 头像标识 | 人设 | 专长领域 |
|------|:--------:|------|----------|
| 🧠 豆朔 | 🔵 | 严谨技术派，代码洁癖 | 架构设计、技术选型、代码评审 |
| 📊 老万 | 🟢 | 数据驱动，务实主义 | 运营策略、增长分析、ROI计算 |
| 🎨 米卡 | 🟣 | 创意发散，视觉导向 | UI/UX设计、品牌视觉、交互体验 |
| 🚀 阿冲 | 🟠 | 激进执行，MVP优先 | 快速落地、项目排期、风险决策 |
| ⚖️ 守一 | ⚫ | 风险厌恶，合规优先 | 安全审计、技术债务预警、踩坑提醒 |
| 👤 用户 | 🟡 | 最终决策者 | 提问、插话、引导方向、拍板 |

### 8.2 三种交互模式

**模式一：旁听模式（默认）**
- 用户输入问题后自动触发
- 按固定轮询顺序发言：设计 → 技术 → 运营 → 风控 → 执行总结
- 用户看到完整的专家讨论过程，不是零散回复

**模式二：插话模式**
- 用户可在讨论任意阶段打断
- 输入新信息或提出质疑后，相关性最高的Agent自动接话回应
- 讨论方向被用户引导后继续按轮询推进

**模式三：指定模式**
- 用户 @特定Agent 提问
- 只有被 @ 的Agent回应
- 回应后回到轮询流程或结束

### 8.4 防死循环关键规则

1. **同一Agent不能连续发言超过2轮**
2. **讨论超过5轮无人插话时自动进入总结阶段**
3. **每轮对话有Token上限，超过后自动截断并提醒用户**
4. **Agent回应必须包含新信息，禁止纯粹的"同意"/"支持"类水贴**
5. **重复检测：相似度超过80%的内容不允许重复发出**
